# 開発ログ: Spicy (辛口献立簿) - UI/UX改善と運用開始フェーズ

## 1. プロジェクト概要
- **目的**: ユーザーの食事記録に対し、AIが「辛口」でコメントする健康管理アプリのUX改善
- **スタック**: Next.js (App Router), Supabase, Vercel, Google Gemini API

## 2. 実装・改善内容
- **UIリフレッシュ**:
    - ヘッダーのコンパクト化とブランド名の変更 ("Spicy")
    - カレンダービューのヒートマップ化（スコアに応じた赤・黄・緑の色分け）
- **機能実装**:
    - SNSシェア機能（一時実装したが「微妙」との判断で削除、判断の迅速さが重要）
    - 過去データのリセットスクリプト (`scripts/clear-data.ts`)
- **トラブルシューティング**:
    - **Supabase Auth**: 新規ユーザー登録時のリダイレクトエラー（`localhost`に戻される問題）。`Site URL` と `Redirect URLs` の設定不備が原因。
    - **Google OAuth**: テストモードにおけるユーザー制限。テストユーザーへの追加で対応。

## 3. 学んだ教訓 (Lessons Learned)
- **Supabase Authの本番設定**:
    - Vercel等にデプロイした際、Supabase側の `Site URL` だけでなく、`Redirect URLs` にも本番ドメイン（ワイルドカード付き `https://.../**`）を追加しないと、新規登録フローで `localhost` に飛ばされる罠がある。
    - 既存ユーザーはセッションが残っているとこの問題に気づきにくいため、必ずシークレットウィンドウ等で新規フローを確認すべき。
- **iOSアプリ化の壁**:
    - Webラッパー（PWA）のままApp Store審査を通すのは極めて困難（Guideline 4.2）。
    - 「Sign in with Apple」と「アカウント削除機能」は必須要件。
    - 今後は「最強のPWA」を目指すか、Capacitor/Expo等でネイティブ化するかの二択。
- **機能の取捨選択**:
    - 実装した機能（シェアボタン）でも、UXに合わなければ即捨てる勇気を持つこと。コードはGitに残るので恐れずに消す。

## 4. セキュリティ・コスト管理メモ
- **セキュリティ**:
    - `clear-data.ts` は強力な権限（SERVICE_ROLE_KEY）を使うため、環境変数の扱いに注意。
    - クライアント側では `NEXT_PUBLIC_SUPABASE_ANON_KEY` のみを使用し、RLSでデータを保護していることを再確認。
- **コスト**:
    - Gemini API: 現在は `flash` モデルを使用しており、無料枠または低コストで運用可能。
    - Vercel: Cron Job (`api/cron`) の実行時間に注意（Hobbyプランは制限がきつい）。現在は `maxDuration = 60` で設定済み。
